# Goal

Modify all three ONCE `run` entrypoints so the copied launcher file is also the
user-editable default profile template:

- `once/clojure/run`
- `once/python/run`
- `once/typescript/run`

# Review notes

- The current `run` files are thin launchers only.
- The active profile currently comes from each implementation's packaged
  `options` module/file (`options/bb`, `once.options.bb`, `bb` from
  `src/once/options.ts`).
- Therefore, putting a profile map in `run` is not enough by itself; the profile
  must be passed into the package/workflow entrypoint, or the packaged `bb`
  profile will still be used.
- Keep secrets out of source. The template should contain placeholders for
  credentials and user-specific values, not real credentials.

# Resolved decision

Use valid example/default values for selector/default/typed fields. Only
user-specific values and secrets should be `"REPLACE_ME"`.

Concretely:

- Use concrete default selectors so the untouched profile has a coherent default
  provider path: `provider-compute = digitalocean`, `provider-smtp = resend`,
  `provider-dns = cloudflare`, `provider-backend = r2`.
- Keep service defaults and typed fields valid, for example SMTP server/port,
  compute names/images/sizes/regions, OCI sizing integers, and the
  `once.applications` list shape.
- Use `"REPLACE_ME"` for credentials, tokens, SSH public keys, cloud resource
  IDs, domains/hosts, and application environment values.
- Use valid default names for non-secret labels, e.g. `profile = "default"`
  and `package = "default"`; users can edit them, but they should not be
  placeholders.

# Proposed behavior

Each `run` file should define one default active profile in that language's
native syntax and pass it to ONCE instead of using the packaged `bb` profile.
The profile should be one map/object with:

- `profile`: `"default"`.
- `params`: one map containing the union of supported ONCE profile parameters
  across all compute, SMTP, DNS, and backend variants, using the value policy
  from the resolved decision above.

The command UX should remain unchanged:

```sh
bb run package validate
python run package validate
node run package validate
```

`BC_PAR_*` environment variable overrides should continue to work after the
run-local profile is introduced.

# Acceptance criteria

- All three `run` files expose the same default-profile keys.
- The run-local profile is actually used by package and individual-tool
  commands.
- Invoking `package validate` against the untouched template reports placeholder
  replacement errors rather than silently using the packaged sample profile.
- No generated `.dist/` files are edited.

# Execution status

Completed.

Implemented local default profile templates in:

- `once/clojure/run`
- `once/python/run`
- `once/typescript/run`

The Python and TypeScript run scripts pass the profile to the packaged CLI via
`ONCE_PROFILE_JSON`; their CLI entrypoints now read that override. The
TypeScript compiled CLI in `dist/` was rebuilt so installed package binaries see
the override too. The Clojure run script passes the profile directly to
`cli/main*`, which now accepts an optional profile argument.

Verification run:

```sh
cd once/clojure && bb run package validate
cd once/python && python3 -m py_compile run src/once/cli.py
cd once/python && python3 run package validate
cd once/typescript && node --check run
cd once/typescript && npm run typecheck
cd once/typescript && npm run build
cd once/typescript && node run package validate
cd once/typescript && node dist/src/cli.js help
```

The untouched templates report `REPLACE_ME` validation errors, confirming that
the run-local profiles are used instead of the packaged sample profiles.
