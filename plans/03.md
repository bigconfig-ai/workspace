### 1. Launcher (`bc-pkg`)

Source of truth: `launcher/typescript/CLAUDE.md`,
`launcher/python/CLAUDE.md`, `launcher/typescript/bin/bc-pkg.js`,
`launcher/python/src/bc_pkg/cli.py`.

Subsections (sidebar items):

- About bc-pkg — what it is, the two equivalent implementations (PyPI
  `bc-pkg`, npm `bc-pkg`), the GitHub-pinned target package model.
- Install & invoke — `uvx bc-pkg ...` (Python tab) and `npx bc-pkg ...`
  (TypeScript tab). For the **Clojure tab**, show how to use bc-pkg to
  bootstrap a Clojure target (`uvx bc-pkg bigconfig-ai/once@clojure ...`
  or `npx bc-pkg bigconfig-ai/once@clojure ...`), explain that bc-pkg
  itself isn't implemented in Clojure but consumes Clojure target
  packages with automatic Babashka + Temurin JDK + git provisioning.
- Spec format — `<owner/repo@ref>` where `ref` is a branch, tag, or full
  40-char SHA.
- Ref resolution — GitHub API, `GITHUB_TOKEN`, 40-char SHA pass-through.
- Target detection — `deps.edn` / `package.json` / `pyproject.toml`
  exclusivity rule.
- On-disk artifacts — the language-native manifests written into cwd
  (`deps.edn` + `bb.edn` for Clojure; `package.json` with `bigconfig`
  block for TS; `pyproject.toml` with `[tool.bigconfig]` block for
  Python). Show actual example output for each.
- Re-entry & validation — repo/ref/sha must match; mismatch is a hard
  error.
- Babashka + JDK cache — cache layout, `BB_VERSION`, `JDK_VERSION`,
  `XDG_CACHE_HOME` / `LOCALAPPDATA`.
- Linux git auto-install — apt/dnf/yum/zypper/pacman/apk, sudo handling.
- Parity guarantee — TS launcher and Python launcher must produce
  equivalent artifacts.

### 2. Once

Source of truth: `once/{typescript,python,clojure}/CLAUDE.md` and
sources (`options`, `package`, `params`, `tools`, `validation`,
`describe`, `cli`, `src/resources/.../tools/`).

Subsections:

- About Once — what it does, the audience (ONCE deployment), how it
  composes BigConfig.
- CLI commands — `package validate|describe|build|create|delete`,
  individual tool runners (`tofu`, `tofu-smtp`, `tofu-dns`,
  `tofu-smtp-post`, `ansible`, `ansible-local`).
- Six-stage create pipeline — diagrammed (tofu → tofu-smtp → tofu-dns →
  tofu-smtp-post → ansible-local → ansible) with what each stage does.
- Delete pipeline — reverse of the four Tofu stages;
  `BC_PAR_COMPUTE_PREVENT_DESTROY=false` override.
- Profiles — `profileAlpha` / `profileBeta` / `profileGamma` /
  `profileNoInfra` (+ Clojure / Python equivalents), how sub-profiles
  (`resend`, `cloudflare`, `r2`, `digitalocean`, `oci`, `hcloud`,
  `noInfraCompute`, `deploy`) compose.
- Active profile binding — `export const bb = profileAlpha` (TS) /
  `bb: Opts = profile_alpha` (Python) / `(def bb profile-alpha)`
  (Clojure).
- Parameter flow — `optsFn` / `opts_fn` / `opts-fn` composition:
  `readBcPars` → `tofuSmtpParams` → `tofuParams`.
- `BC_PAR_*` env vars — uppercased, hyphens/dots become underscores,
  `.envrc.private` convention.
- Validation step — schema, tool/credential/image/ssh-agent checks,
  `validateReport` / `validate_report` / `validate-report` pure
  builders.
- Describe step — provider list + SSH reachability + deployed apps
  table (image, tag, running digest, registry digest, update-available).
- Template structure — `src/resources/.../tools/`, file content uses
  `<{ var }>`, directory selection uses `{{ var }}` so Ansible's `{{ }}`
  passes through.
- Plugin system — `render-tofu-backend` plugin, S3 / R2 / local backend
  switching.
- Build artifact — `.dist/profile-alpha-<hash>/` parity goal across the
  three implementations.

### 3. BigConfig

Source of truth: `big-config/{typescript,python,clojure}/CLAUDE.md` and
sources (`core`, `workflow`, `pluggable`, `render`, `run`, `git`,
`lock`, `unlock`, `step-fns`, `selmer-filters`, `big-tofu/`).

Subsections:

- About BigConfig — workflow + render engine; consumed by Once.
- Concepts — the `opts` map threading model.
- Reserved opts keys — `big-config/exit`, `big-config/err`,
  `big-config/stack-trace`, `big-config.workflow/steps`,
  `big-config.workflow/params`, etc. Verbatim string keys, not
  converted to snake_case or camelCase.
- Workflow primitives — `createWorkflow` / `workflow` /
  `create-workflow`; `ok`; `choice`; `stepFn` / `step_fn` / `step-fn`.
- Composition layer — `createWorkflowStar` / `workflow_star` /
  `create-workflow-star`; `runSteps` / `run_steps` / `run-steps`;
  CLI argument parsing.
- Renderer — Selmer with custom delimiters; rendering `src/resources/`
  into `.dist/`; `:transform [[<provider> delimiters]]`.
- Shell runner — the `runner` seam (`run.runner`, `big-config.run/runner`),
  test injection.
- Locking — Git-tag pessimistic locking; `unlock` force-release.
- Git helpers.
- Pluggable steps — step registry for plugins.
- Step-fn middleware.
- Selmer filters — BigConfig's custom filters.
- `big-tofu` — construct helpers and references.
- CLI — `big-config` / `uv run big-config` / `clojure -M:run`.

### 4. Selmer

Source of truth: `selmer/{typescript,python,clojure}/CLAUDE.md` and
sources.

Subsections:

- About Selmer — Django-inspired template engine, Clojure upstream
  (`yogthos/Selmer`), TS and Python stdlib-only ports.
- Quick start — render a template with a context map.
- Template language — variables (`{{ name }}`, dotted paths, namespaced
  keys), filters (`|upper`, chainable), tags.
- Public API — `render`, `renderFile` / `render_file` / `render-file`,
  `parse*`, `renderTemplate` / `render_template`, `addFilter` /
  `add_filter` / `add-filter!`, `addTag`, `cacheOn`/`cacheOff`,
  `validateOn`/`validateOff`, `setResourcePath`, `knownVariables`.
  Mutating-name convention table:
  - Clojure: `add-filter!`, `cache-on!`, …
  - Python: `add_filter` and `add_filter_bang`, …
  - TypeScript: `addFilter` and `addFilter$` (both forms exported), …
- Built-in tags — `{% if %}`, `{% for %}`, `{% block %}`, `{% extends %}`,
  `{% include %}`, `{% comment %}`, `{% cycle %}`, `{% safe %}`, etc.
- Built-in filters — alphabetical reference of the bundled filters.
- Custom delimiters — `tagOpen` / `tagClose` / `filterOpen` /
  `filterClose` opts (TS), `tag_open` / `tag_close` / `filter_open` /
  `filter_close` (Python), `:tag-open` / `:tag-close` / `:filter-open` /
  `:filter-close` (Clojure). Explain BigConfig's use of `<{ ... }>` so
  Ansible's `{{ ... }}` passes through.
- Extensibility — registering custom filters and tags.
- Caching & validation — template cache toggle, validation toggle,
  `SelmerValidationError`.
- Internal seams — `safe`, scanner, node tree, template inheritance
  resolver, middleware integration helpers.
