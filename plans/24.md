# Plan 24 — Skills: scaffold Python and Clojure BigConfig packages from scratch

## Goal

Add two new skills to the `skills/` repository — `create-py-bigconfig-package` and
`create-clj-bigconfig-package` — that are the Python and Clojure twins of the existing
`create-ts-bigconfig-package`. Each scaffolds the smallest *real* BigConfig package in
its language: depends only on the `big-config` SDK (not `once`), renders one template
through a single `hello` stage, exposes `validate` + `build` verbs, ships a
launcher-friendly root `run`, and builds/tests green. The author then grows it.

This mirrors Plan 23 (which built the TypeScript skill) one-for-one in two more
languages, so the three skills stay behaviourally consistent.

## Decisions from user

- **Scope/model:** minimal hello-stage parity with the TS skill — big-config SDK only,
  one `hello` render stage, `validate` + `build` verbs. No `once` dependency, no
  tofu/ansible pipeline.
- **Verification:** for each skill, generate a throwaway sample, run it green
  end-to-end, then delete it (toolchain confirmed available: `uv` 0.11, `python` 3.12,
  `bb` 1.12, `clojure` 1.12, `java` 25, `tofu` 1.11, `node` 25).
- **Repo docs:** add both skills to `skills/README.md`'s table and extend
  `skills/CLAUDE.md`'s token-convention section to document the Python module-name and
  Clojure underscore-munging gotchas.

## Non-goals

- Do not change the TS skill, the `big-config`/`selmer`/`once`/`walter` leaves, the
  launchers, or the package-spec page.
- Do not generate `once`/`walter`-scale packages (no `once` dep, no multi-stage
  create/delete pipeline, no provider profiles).
- No real credentials anywhere — placeholders only (`REPLACE_ME`).
- Do not publish or commit generated packages.

## Source material to mirror

- The reference skill: `skills/create-ts-bigconfig-package/SKILL.md`, its
  `templates/`, and `reference/checklist.md`. Keep the same six-step procedure shape
  (collect inputs → derive tokens → copy/rename → substitute → verify → report) and the
  `.tmpl`-suffix = contains-tokens convention.
- The repo conventions: `skills/CLAUDE.md` (template-token rules, "what the generated
  package looks like", "verifying a change").
- **Python shape** (model on the SDK-only consumer pattern, stripped to one stage):
  `once/python/` and `walter/python/` — `pyproject.toml` (hatchling, `force-include`
  for `src/resources`, `[project.scripts]`, `big-config` git dep), `run` (python3
  shebang + `sys.path` setup + `DEFAULT_PROFILE` + `main(argv, profile)`),
  `src/<module>/{__init__,__main__,cli,package,tools,options,params,validation,interop}.py`
  + `py.typed`, `tests/`. The interop/validation alias handling already carries the
  Plan-23 fix (`read_bc_pars` wraps `sync_aliases`).
- **Clojure shape** (model on `once/clojure`, which depends only on the SDK):
  `once/clojure/deps.edn` (`:paths ["src/clj" "src/resources"]`, SDK as
  `io.github.amiorin/big-config {:git/sha …}` with a commented `:local/root
  "../../big-config/clojure"`), `bb.edn` (`{:deps {io.github.<owner>/<pkg>
  {:local/root "."}}}`), `run` (`#!/usr/bin/env bb`, requires `big-config.render`,
  `big-config.workflow`, the package `cli` ns, a `default-profile` with
  `::render/profile` + `::workflow/params`, calls `cli/main`), `src/clj/<ns-as-path>/…`,
  `src/resources/<ns-as-path>/…`, `test/clj/…` run via `clojure -M:test`.
- Engine APIs are identical across languages (opts threading, reserved namespaced keys,
  `workflow`/`run-steps`/`create-workflow-star`/`prepare`, render templates, runner
  seam) — see each leaf's `CLAUDE.md`.

## Generated package design (both languages)

Same as the TS starter, translated idiomatically:

- One `hello` stage that renders a single template directory
  (`…/tools/hello/greeting.txt` containing `Hello, {{ name }}!`) into
  `.dist/<profile>-<hash>/…` via the SDK renderer + `prepare` + run-steps.
- `validate` verb: a pure report (flags blank / `REPLACE_ME` `package` and `name`
  params, honours `BC_PAR_*`) wired through the SDK validate-fn reserved key; `build`
  verb: render the `hello` stage via the SDK build-fn / workflow-star.
- A single default profile (`bb`) with kebab-case params `package` and `name`
  (`name` default `world`), and a matching editable `DEFAULT_PROFILE` in `run`.
- One test that builds/loads the CLI, runs `package build` in a temp dir, and asserts
  the rendered `greeting.txt` contains `Hello, world!`; plus a `validate` unit test
  (passes clean; fails when `BC_PAR_NAME=REPLACE_ME`).

```
    │   └── resources/tools/hello/greeting.txt   # verbatim; relocate under ns path
    └── tests/
        ├── test_build.py.tmpl
        └── test_validation.py.tmpl
```

### Python tokens

| Token | Meaning | Example (name `widget`, owner `bigconfig-ai`) |
|---|---|---|
| `__PKG_NAME__` | PyPI / dir / CLI name (kebab ok) | `widget` |
| `__PKG_MODULE__` | importable module name (snake_case; hyphens → `_`) | `widget` (`my-widget` → `my_widget`) |
| `__PKG_OWNER__` | GitHub owner | `bigconfig-ai` |
| `__PKG_NAMESPACE__` | `io.github.<owner>.<name>` (engine step names) | `io.github.bigconfig-ai.widget` |
| `__PKG_NS_PATH__` | namespace as path, hyphens kept (resources) | `io/github/bigconfig-ai/widget` |
| `__PKG_DESC__` | one-line description | `A minimal BigConfig package.` |
| `__BIG_CONFIG_REF__` | dep spec | `big-config @ git+https://github.com/bigconfig-ai/big-config.git@<sha>` |

`__PKG_MODULE__` is the load-bearing extra vs TS: `[project.scripts]`,
`[tool.hatch.build.targets.wheel].packages = ["src/<module>"]`, the `src/<module>/`
dir, and every intra-package import use it. SDK ref default = current sibling-leaf SHA
(`grep` from `once/python/pyproject.toml`); local-dev alternative is
`[tool.uv.sources] big-config = { path = "../../big-config/python", editable = true }`.

### Python verification

`uv sync` → `uv run pytest -q` → `uv run python run package validate` (exit 0) →
`uv run python run package build` (renders `.dist/…/hello/greeting.txt`) →
`BC_PAR_NAME=REPLACE_ME uv run python run package validate` (must fail, exit 1).
Launcher runtime command for this target is `uv run python run …`.

### Clojure tokens

| Token | Meaning | Example (name `widget`, owner `bigconfig-ai`) |
|---|---|---|
| `__PKG_NAME__` | package / CLI name (kebab ok) | `widget` |
| `__PKG_OWNER__` | GitHub owner | `bigconfig-ai` |
| `__PKG_NAMESPACE__` | dotted namespace (ns decls, engine step names) | `io.github.bigconfig-ai.widget` |
| `__PKG_NS_SRC_PATH__` | **source** path: hyphens → `_` (Clojure munging) | `io/github/bigconfig_ai/widget` |
| `__PKG_NS_RES_PATH__` | **resources** path: hyphens kept | `io/github/bigconfig-ai/widget` |
| `__PKG_COORD__` | bb.edn/launcher coordinate `io.github.<owner>/<name>` | `io.github.bigconfig-ai/widget` |
| `__PKG_DESC__` | one-line description | `A minimal BigConfig package.` |
| `__BIG_CONFIG_REF__` | SDK dep map value | `{:git/sha "<sha>"}` under `io.github.amiorin/big-config` |

**The load-bearing Clojure gotcha:** namespace declarations use hyphens
(`io.github.bigconfig-ai.widget.tools`) but the source file lives at the underscore path
(`src/clj/io/github/bigconfig_ai/widget/tools.clj`); a hyphenated package name munges in
the path too (`my-widget` → `my_widget`). Resources keep hyphens. `__PKG_NS_SRC_PATH__`
and `__PKG_NS_RES_PATH__` must be derived independently and used in the right places, or
namespaces won't load / templates won't be found. SDK ref default = current
`io.github.amiorin/big-config` `:git/sha` from `once/clojure/deps.edn`; commented
`:local/root "../../big-config/clojure"` line included for local dev. Include the
baseline runtime deps the SDK needs at render time (e.g. `org.clojure/clojure`,
`babashka/process`, `cheshire`) — confirm the minimal set during implementation.

### Clojure verification

`clojure -M:test` (cognitect test-runner) → `bb run package validate` (exit 0) →
`bb run package build` (renders `.dist/…/hello/greeting.txt`) →
`BC_PAR_NAME=REPLACE_ME bb run package validate` (must fail, exit 1). Launcher runtime
command for this target is `bb run …`.

## SKILL.md procedure (both skills)

Mirror the TS skill's six steps, adjusted per language:

1. **Collect inputs** — package name, GitHub owner, description, target directory
   (default `<name>/<lang>` workspace sibling), SDK source (SHA pin default vs local
   path).
2. **Derive tokens** — per the tables above (note the module-name / dual-path
   derivations).
3. **Copy templates + rename** — list every rename explicitly (incl. relocating
   `templates/src/resources/tools/` under the namespace resource path, and the source
   `pkg/` dir under the module/ns source path), dropping `.tmpl` suffixes.
4. **Substitute tokens** — exact, with the language-specific warnings.
5. **Install / test / run** — the verification commands above; the `BC_PAR_NAME`
   override case must fail.
6. **Report** — print the checklist confirmed; explain how to grow the package and make
   it `bc-pkg`-installable (`uvx bc-pkg <owner>/<name>@<ref> package build` /
   `npx bc-pkg …` per spec; local path form for dev).

## Implementation steps

1. Build a concrete Python sample (`widget` / `bigconfig-ai`) by hand, iterate until
   `uv` install/test/run is green and the `BC_PAR_NAME` override fails validation
   (carrying the Plan-23 alias fix forward). Tokenize verified files into
   `create-py-bigconfig-package/templates/`; write its SKILL.md + checklist.
2. Re-instantiate the Python templates with *fresh, hyphenated* inputs (e.g.
   `data-box` / `acme`) per the procedure and confirm green; delete the throwaway.
3. Repeat 1–2 for Clojure (`bb`/`clojure`), paying special attention to the dual path
   convention; re-instantiate with a hyphenated name to prove the munging.
4. Update `skills/README.md` and `skills/CLAUDE.md`.
5. Leave changes staged/uncommitted (per repo policy) unless asked to commit.

## Acceptance criteria

- Two new skill directories exist, each with valid `SKILL.md` frontmatter
  (`name`/`description`), `templates/`, and `reference/checklist.md`.
- Running each skill against sample inputs produces a package that, in its generated dir:
  - **Python:** `uv sync`, `uv run pytest -q` pass; `uv run python run package validate`
    exits 0; `uv run python run package build` emits `.dist/…/hello/greeting.txt`
    containing `Hello, world!`; `BC_PAR_NAME=REPLACE_ME … validate` exits 1.
  - **Clojure:** `clojure -M:test` passes; `bb run package validate` exits 0; `bb run
    package build` emits the rendered greeting; `BC_PAR_NAME=REPLACE_ME bb run package
    validate` exits 1.
- Each generated package satisfies its conformance checklist: exactly one root manifest
  (`pyproject.toml` / `deps.edn`), declared name, root `run` forwarding args, resources
  shipped in the installable artifact, no secrets.
- Hyphenated-name re-instantiation works for both (module snake_case / Clojure
  underscore munging proven), with zero leftover tokens.
- The three skills are behaviourally consistent (same verbs, same `hello` output shape,
  same `BC_PAR_*` semantics).
- `skills/README.md` and `skills/CLAUDE.md` document the two new skills.
- Verification output is recorded during implementation; throwaways removed afterward.
- No changes outside `skills/` (and the temporary sample dirs, which are deleted).
