# Plan 27 — Make scaffolded `hello render` use the package build output directory

## Goal

Update the three BigConfig package scaffold skills (`create-ts-bigconfig-package`,
`create-py-bigconfig-package`, and `create-clj-bigconfig-package`) so a generated
minimal package writes the same top-level `.dist/<profile>-<hash>/` directory when
running either:

- `package build`
- `hello render`

The current templates let `package build` go through the package `workflow_star`,
which computes `.dist/<profile>-<hash>/`, while the direct `hello render` tool path
renders as a raw tool workflow and can write a different top-level directory under
`.dist/`.

## Source guidance reviewed

- `/home/ubuntu/code/bigconfig/skills/CLAUDE.md`
- `/home/ubuntu/code/bigconfig/once/clojure/CLAUDE.md`
- Once Clojure reference shape:
  - `once/clojure/src/clj/io/github/bigconfig_ai/once/package.clj`
  - `once/clojure/src/clj/io/github/bigconfig_ai/once/tools.clj`
  - `once/clojure/src/clj/io/github/bigconfig_ai/once/cli.clj`
- BigConfig workflow prefix logic:
  - `big-config/{clojure,python,typescript}/.../workflow.*`

## Confirmed decisions

1. Scope is only the three skill templates/docs under `skills/`; do not update any
   already-generated packages in the workspace.
2. `hello render` must use the active package profile (`bb` / root `run` default
   profile) and must honor `BC_PAR_*` overrides through the same `opts_fn` /
   `opts-fn` path used by `package build`.
3. Acceptance compares only the exact top-level directory name under `.dist/` (for
   example `.dist/widget-<hash>/`). The full relative path to `greeting.txt` below
   that directory does not need to match.
4. `package create` should continue to render to the same directory as `package build`
   (already intended by the templates); this change is primarily about aligning the
   direct `hello render` command.

## Implementation plan

### 1. Add package-build-prefix helper in each generated package template

For each language's `package` template, expose a small helper that mirrors the prefix
calculation used by the package `build` workflow:

- TypeScript: `skills/create-ts-bigconfig-package/templates/src/pkg/package.ts.tmpl`
  - Export a helper such as `helloPackageOpts(opts: Opts): Opts` or
    `packageToolOpts(opts: Opts): Opts`.
  - It should:
    1. normalize friendly aliases with `toBcOpts`,
    2. compute `newPrefix(selectGlobals(opts), RENDER_PIPELINE_START)`,
    3. merge those globals back into the active opts,
    4. apply `optsFn` so direct tool renders honor `BC_PAR_*`.
- Python: `skills/create-py-bigconfig-package/templates/src/pkg/package.py.tmpl`
  - Add the equivalent helper using `bc_workflow.new_prefix` and
    `bc_workflow.select_globals`.
- Clojure: `skills/create-clj-bigconfig-package/templates/src/clj/pkg/package.clj.tmpl`
  - Add the equivalent helper using `workflow/new-prefix`,
    `workflow/select-globals`, and `params/opts-fn`.

Keep the helper package-level rather than tool-level because it needs the package
build first-step (`package/start-render`) to reproduce the same hash and should not
make `tools` depend on `package`.

### 2. Route direct `hello render` through the helper

Update each CLI template so the direct `hello` command passes the package-aligned opts
to the tool workflow:

- TS: `src/cli.ts.tmpl`
  - Import the new helper from `./__PKG_NAME__/package.js`.
  - Change `helloStar(rest, opts)` to `helloStar(rest, packageToolOpts(opts))`.
- Python: `src/pkg/cli.py.tmpl`
  - Import the helper from `.package`.
  - Change `hello_star(rest, active_profile)` to use the package-aligned opts.
- Clojure: `src/clj/pkg/cli.clj.tmpl`
  - Use the package helper instead of calling only `params/opts-fn` in the `hello`
    branch.
  - Remove the now-unused direct `params` require if it becomes unnecessary.

### 3. Strengthen generated tests to compare output directories

Update the scaffolded package tests in all three skills:

- TS: `templates/test/build.test.ts.tmpl`
- Python: `templates/tests/test_build.py.tmpl`
- Clojure: `templates/test/clj/pkg/package_test.clj.tmpl`

Add/adjust assertions so they:

1. run `package build` in a clean temp working directory,
2. capture the single top-level directory created under `.dist/`,
3. run `hello render` in a clean temp working directory,
4. capture the single top-level directory created under `.dist/`,
5. assert the two top-level directory names are identical,
6. assert `greeting.txt` exists somewhere under that top-level directory and contains
   `Hello, world!`.

Do not require the full relative path to `greeting.txt` below the top-level `.dist/`
directory to match; only the top-level package output directory is part of this plan's
acceptance contract.

Also update `skills/CLAUDE.md` if needed so the shared verification summary includes

### 5. Verify by scaffolding all three languages

Because `skills/` itself has no build system, verify by generating sample packages from
the templates, preferably with a hyphenated package name (for example `data-box`) to
exercise identifier/path derivations.

Run each generated package's checks:

- TypeScript:
  - `npm install`
  - `npm run build`
  - `npm run typecheck`
  - `npm test`
  - `node run package build`
  - `node run hello render`
- Python:
  - `uv sync`
  - `uv run pytest -q`
  - `uv run python run package build`
  - `uv run python run hello render`
- Clojure:
  - `clojure -M:test`
  - `bb run package build`
  - `bb run hello render`

Manual/automated acceptance: after deleting `.dist/` between runs, `package build` and
`hello render` must create the same single top-level directory under `.dist/`, and the
rendered `greeting.txt` must contain `Hello, world!`.
