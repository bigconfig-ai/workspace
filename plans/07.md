# Goal

Implement `bc-pkg` launchers for TypeScript/npm and Python/PyPI with the same
user-facing behavior.

# User experience

```sh
uvx bc-pkg <owner/repo@ref> package validate
npx bc-pkg <owner/repo@ref> package validate

# after the first run, the repo spec is remembered in the current directory
uvx bc-pkg package validate
npx bc-pkg package validate
```

`ref` may be either a branch name or a commit SHA. No separate SHA notation is
required; use the same `@ref` slot:

```sh
npx bc-pkg bigconfig-ai/once@typescript package validate
npx bc-pkg bigconfig-ai/once@2f4e8c0d0b4c4b8f0c3a9f6e2a1b5c7d8e9f0123 package validate
```

The launcher should not implement a `validate` shortcut for now. Commands are
forwarded as written, so the supported syntax is `package validate`.

# Current understanding

- `bc-pkg` is both an npm package and a PyPI package.
- `bc-pkg` is a launcher/bootstrapper: it creates or reuses a BigConfig CLI in
  the current directory, then forwards the command to that CLI.
- A BigConfig package is selected by a GitHub repo + ref, written as
  `<owner/repo@ref>`.
- The selected ref determines the target implementation language. The launcher
  should infer the language from the branch/SHA contents, not from the launcher
  used (`npx` vs `uvx`).
- Both launcher implementations must support packages implemented in Clojure,
  Python, and TypeScript.
- There is no Clojure implementation of the launcher itself.
- For Clojure target packages, the launcher should keep the current behavior:
  install/check JDK, Babashka, and git as needed, then run the Babashka CLI.
- For TypeScript target packages, require Node/npm to already exist.
- For Python target packages, require Python/uv to already exist.
- The TypeScript launcher needs to be adapted to work with the `once` package.
- The Python launcher should be ported from the TypeScript launcher after the
  TypeScript behavior is correct.

# Existing state to account for

- `launcher/typescript` is on the `typescript` branch.
- `launcher/python` is on the `python` branch.
- Both currently contain the same JavaScript `bin/bc-pkg.js`, which bootstraps
  Babashka/JDK/git and `bb.edn`; this is effectively the Clojure-target behavior
  and not yet the final TypeScript/Python launcher behavior.
- `once/typescript/run` and `once/python/run` have been added so the launcher
  can copy the package's root `run` file for those target languages, matching
  the Clojure branch's root `run` file pattern.

# Resolved decisions

- Use language-native files for persisted launcher metadata:
  - Clojure: `deps.edn`
  - TypeScript: `package.json`
  - Python: `pyproject.toml`
- Detect the target language from branch/SHA contents by looking for:
  - Clojure: `deps.edn`
  - TypeScript: `package.json`
  - Python: `pyproject.toml`
- Create the local CLI by copying only the target package's root `run` file into
  the current directory, plus writing/updating the language-native manifest.
  For Clojure targets, also write a minimal `bb.edn` runtime file because
  Babashka scripts load dependencies from `bb.edn`, while launcher metadata
  stays in `deps.edn`.
- Treat the run files as equivalent entrypoints:
  - Clojure target: `bb run ...`
  - TypeScript target: `node run ...`
  - Python target: `uv run python run ...`
- Do not add a `validate` shortcut yet. Users must run `package validate`.
- Do not bootstrap Node/npm, Python, or uv. They must already be installed when
  needed by the selected target language.
- Pin the resolved SHA on first run and keep using that SHA on later runs.
- Accept a commit SHA in the same notation as a branch: `<owner/repo@ref>`.
- If an initialized directory is invoked with a different `<owner/repo@ref>` or
  a different pinned SHA than the one in the native metadata file, exit with an
  error. Do not update or migrate implicitly.

# Metadata shape

Use this metadata shape unless implementation constraints force a small change.

## Clojure: `deps.edn`

```clojure
{:deps {io.github.<owner>/<repo> {:git/url "https://github.com/<owner>/<repo>.git"
                                  :git/sha "<sha>"}}
 :bigconfig/repo "<owner>/<repo>"
 :bigconfig/ref "<original-ref>"
 :bigconfig/sha "<sha>"
 :bigconfig/language "clojure"
 :bigconfig/run "run"}
```

## TypeScript: `package.json`

```json
{
  "type": "module",
  "scripts": {
    "run": "node run"
  },
  "dependencies": {
    "<package-name>": "github:<owner>/<repo>#<sha>"
  },
  "bigconfig": {
    "repo": "<owner>/<repo>",
    "ref": "<original-ref>",
    "sha": "<sha>",
    "language": "typescript",
    "run": "run",
    "packageName": "<package-name>"
  }
}
```

## Python: `pyproject.toml`

```toml
[project]
dependencies = [
  "<package-name> @ git+https://github.com/<owner>/<repo>.git@<sha>",
]

[tool.bigconfig]
repo = "<owner>/<repo>"
ref = "<original-ref>"
sha = "<sha>"
language = "python"
run = "run"
package-name = "<package-name>"
```

# Proposed contract

1. First run with `<owner/repo@ref>`:
   - parse the repo spec;
   - if `ref` is a branch, resolve and pin the branch HEAD SHA;
   - if `ref` is a SHA, validate/normalize it and pin that SHA directly;
   - fetch enough content at the pinned SHA to detect the target language;
   - copy the target package's root `run` file into the current directory;
   - write/update the language-native manifest with the package dependency and
     `bigconfig` metadata;
   - install/sync dependencies for the selected target language when needed
     (`npm install` for TypeScript, `uv sync` for Python; Clojure continues to
     use the existing Babashka/JDK/git bootstrap path and a generated `bb.edn`);
   - run the requested command through the copied `run` file.
2. Later runs without `<owner/repo@ref>`:
   - read the persisted metadata from the language-native file in the current
     directory;
   - reuse the previously selected package and pinned SHA;
   - run the requested command through the copied `run` file.
3. Passing `<owner/repo@ref>` again in a directory that already has launcher
   metadata must validate that it resolves to the same repo/ref/SHA metadata.
   If it does not match, exit with an error.

# Ref/SHA notation

- The canonical input is `<owner/repo@ref>`.
- `ref` can be a branch name or a commit SHA.
- No special prefix is required for SHA input.
- Implementation recommendation:
  - if `ref` is a full 40-character hex commit SHA, treat it as an explicit SHA;
  - otherwise resolve it through GitHub as a branch/ref and store the resulting
    full SHA;
  - always persist the full pinned SHA so later runs never drift.

# Implementation plan

1. Implement language detection from branch/SHA contents using the manifest
   files above.
2. Implement copying the target package's root `run` file.
3. Implement native manifest generation/update with the metadata shape above.
4. Implement mismatch detection: an existing initialized directory with a
   different repo/ref/SHA is a fatal error.
5. Implement the TypeScript/npm launcher first and verify it against `once`.
6. Port the TypeScript behavior to the Python/PyPI launcher.
7. Keep Clojure-target support by reusing the current Babashka/JDK/git bootstrap
   behavior, adjusted to `deps.edn` metadata and the copied `run` file.
8. Add parity tests so `npx bc-pkg ...` and `uvx bc-pkg ...` behave the same.
