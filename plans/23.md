# Plan 23 — Skill: scaffold a TypeScript BigConfig package from scratch

## Goal

Create a **Claude Code skill** that scaffolds a new, minimal, launcher-conformant
**TypeScript BigConfig package** from scratch. Invoking the skill produces a fresh
package directory that depends only on the `big-config` TypeScript SDK, builds, tests
green, and runs through the `bc-pkg` launcher contract documented in the BigConfig
Package Spec (`once-bigconfig/main/src/pages/package-spec.html`, plan 22).

The generated package is the *smallest thing that is a real package*: one example
render stage, `validate` + `build` verbs, a launcher-friendly root `run`, one packaged
template resource, and a passing test — a seed an author then grows, not a clone of
`once`/`walter`.

## Decisions from user

- **Skill format/location:** a real Claude Code skill (a `SKILL.md` with
  `name`/`description` frontmatter plus supporting files) living in a top-level
  `skill/` folder at the workspace root (`/home/ubuntu/code/bigconfig/skill/`), per
  the original stub's line 2.
- **Scaffold scope:** minimal launcher-conformant starter — depends on the
  `big-config` SDK only (not `once`), with a single example stage/tool.
- **Generation strategy:** the skill ships concrete starter files under
  `skill/templates/`; the procedure copies them into the new package directory and
  substitutes package name / owner / namespace tokens.
- **Verification:** acceptance includes generating a sample package and confirming
  `npm install`, `npm run build`, `npm test`, and `node run package validate` /
  `node run package build` all succeed.

## Non-goals

- Do not generate Python or Clojure packages (TypeScript only). A later plan can add
  sibling generators.
- Do not generate `once`/`walter`-scale packages (no tofu/ansible six-stage pipeline,
  no `once` dependency, no provider profiles).
- Do not change the `big-config` SDK, `selmer`, the launchers, or the package-spec page.
- Do not embed real credentials anywhere; placeholders only (`REPLACE_ME`).
- Do not publish the generated package or commit anything (per repo Git policy).

## Source material to mirror

The starter files are distilled from the existing TypeScript leaves — read these for
exact shapes before authoring templates:

- Manifest/build: `once/typescript/package.json`, `tsconfig.json`, `vitest.config.ts`.
- Launcher entrypoint: `once/typescript/run`, `walter/typescript/run` (copied by
  `bc-pkg`; carries an editable default profile, no secrets).
- CLI shape: `once/typescript/src/cli.ts` (exported `main(argv, opts)`, `isMainModule`
  guard, help text).
- Minimal package workflow + a single render stage: `walter/typescript/src/walter/`
  (`tools.ts` is the cleanest one-stage render example: `prepare` + `RENDER_TEMPLATES`
  + `runSteps`; `package.ts` composes verbs via `createWorkflow`/`createWorkflowStar`,
  `registerWorkflow`, and the `WF_VALIDATE_FN` / `WF_BUILD_FN` reserved keys).
- Resource/template convention: `walter/typescript/src/resources/io/github/bigconfig-ai/walter/tools/...`
  (templates under `src/resources/<reverse-dns-namespace>/tools/...`, included in the
  published artifact via `package.json` `files`).
- Conformance contract: plan 22 (`plans/22.md`) and `package-spec.html` — the generated
  package must satisfy "exactly one root manifest", "root `run` exists", declared
  package `name`, resources shipped in `files`, `run` forwards argv + safe defaults.

## Deliverable layout

```
skill/
├── SKILL.md                       # name + description frontmatter; the procedure
├── templates/                     # starter files copied into the new package
│   ├── package.json.tmpl
│   ├── tsconfig.json
│   ├── vitest.config.ts
│   ├── run.tmpl
│   ├── .gitignore
│   ├── README.md.tmpl
│   ├── src/
│   │   ├── cli.ts.tmpl
│   │   └── pkg/
│   │       ├── options.ts.tmpl    # default profile (`bb`) + params
│   │       ├── params.ts.tmpl     # BC_PAR_* overrides → opts
│   │       ├── tools.ts.tmpl      # one render stage (the example tool)
│   │       ├── validation.ts.tmpl # pure validate report
│   │       └── package.ts.tmpl    # validate/build verbs + *Star CLI wrapper
│   │   └── resources/<ns-path>/tools/hello/<{ name }>.txt.tmpl  # sample template
│   └── test/
│       └── build.test.ts.tmpl     # builds, runs `package build`, asserts .dist output
└── reference/
    └── checklist.md               # conformance checklist lifted from package-spec
```

(Token-bearing files use the `.tmpl` suffix so the procedure knows which to substitute;
files with no tokens — `tsconfig.json`, `vitest.config.ts`, `.gitignore` — are copied
verbatim. Final filenames drop `.tmpl`.)

## Substitution tokens

Distinct, collision-free tokens (avoid `{{ }}` / `<{ }>`, which are live template
syntax in resources):

| Token | Meaning | Example |
|---|---|---|
| `__PKG_NAME__` | npm package + CLI name | `widget` |
| `__PKG_OWNER__` | GitHub owner for the SDK dep / namespace | `bigconfig-ai` |
| `__PKG_NAMESPACE__` | reverse-DNS namespace for steps + resource path | `io.github.bigconfig-ai.widget` |
| `__PKG_NS_PATH__` | namespace as a path | `io/github/bigconfig-ai/widget` |
| `__PKG_DESC__` | one-line description | `My BigConfig package.` |
| `__BIG_CONFIG_REF__` | pinned SDK GitHub SHA (or `file:` local override) | `github:bigconfig-ai/big-config#<sha>` |

The skill defaults `__BIG_CONFIG_REF__` to the current SHA pinned in
`once/typescript/package.json` (read at scaffold time) so the seed resolves; it also
documents swapping to `file:../../big-config/typescript` for local SDK dev.

## SKILL.md procedure (what the skill tells Claude to do)

1. Collect inputs (ask the user, with defaults): package name, GitHub owner,
   description, target directory (default `<name>/typescript` sibling, or a path the
   user gives), and whether to pin the SDK by SHA (default) or `file:` local path.
2. Derive `__PKG_NAMESPACE__` / `__PKG_NS_PATH__` from owner + name.
3. Create the target directory; copy `templates/` into it, renaming `*.tmpl` →
   final names and moving the sample resource into `src/resources/__PKG_NS_PATH__/...`.
4. Substitute all tokens in the copied files.
5. `npm install`, `npm run build`, `npm test`.
6. `node run package validate` and `node run package build`; confirm `.dist/...` output
   appears and is git-ignored.
7. Print the conformance checklist (`reference/checklist.md`) with each item checked,
   and next steps (add stages, edit the profile in `run`, publish via a GitHub SHA so
   `npx bc-pkg <owner>/<name>@<ref> package build` works).

## Minimal package design (the generated artifact)

- **One stage**: a `hello` render tool (`tools.ts`) that renders the single template
  resource into `.dist/__PKG_NAME__-<hash>/` via the SDK renderer (`RENDER_TEMPLATES`
  + `prepare` + `runSteps`), modeled on `walter/typescript/src/walter/tools.ts`.
- **Verbs**: `validate` (pure report from `validation.ts`) and `build` (run the render
  stage), wired through the SDK reserved keys `WF_VALIDATE_FN` / `WF_BUILD_FN` and
  exposed as `pkgStar(argv, opts)`; `cli.ts` `main(argv, opts)` dispatches
  `package <verb>` and the `validate` shortcut, mirroring `once/typescript/src/cli.ts`.
- **Profile**: a single `bb` default profile in `options.ts` with a couple of
  kebab-case params (e.g. `name`, `package`) and a matching editable `DEFAULT_PROFILE`
  in `run`.
- **Test**: `build.test.ts` builds the CLI and runs `package build` in a temp dir,
  asserting the rendered file exists with expected content (process-light, no network).

## Acceptance criteria

- `skill/SKILL.md` exists with valid `name` + `description` frontmatter and the
  step-by-step procedure above; templates and reference checklist are present.
- Running the skill against sample inputs (e.g. name `widget`, owner `bigconfig-ai`)
  produces a package where, in that generated dir:
  - `npm install` succeeds (SDK resolves from the pinned ref);
  - `npm run build` and `npm test` pass;
  - `node run package validate` and `node run package build` succeed and emit a
    rendered artifact under `.dist/`.
- The generated package satisfies the package-spec conformance checklist: exactly one
  root manifest (`package.json`), declared `name`, root `run` present and forwarding
  argv, `src/resources` + built `dist` listed in `files`, no secrets in any file.
- No real credentials anywhere; all sensitive fields are `REPLACE_ME`.
- The skill does not modify the SDK, launchers, package-spec page, or any existing leaf;
  it only writes under `skill/` and (when run) the new package directory.
- Verification is recorded by actually generating a throwaway sample package during
  implementation and pasting the passing command output; the throwaway is then removed.
