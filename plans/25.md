# Plan 25 — complete package CLI help, expose workflow verbs, remove bare `validate`

## Goal

Make the `once` and `walter` CLI help and command parsing consistent with the
BigConfig SDK workflow surface across Clojure, Python, and TypeScript.

Required outcomes:

- Write complete CLI help for every supported package verb.
- Ensure package workflows expose these verbs:
  - `validate`
  - `describe`
  - `build`
  - `create`
  - `delete`
  - `lock`
  - `git-check`
  - `git-push`
  - `unlock-any`
- Make Walter gain a real `describe` workflow step in all three languages.
- Show that `lock`, `git-check`, `git-push`, and `unlock-any` are usable both:
  - as package workflow steps, e.g. `package git-check lock build unlock-any`;
  - inside individual tool workflows, e.g. `tofu git-check lock render ...`.
- Remove only the bare validation shortcuts:
  - Clojure: `bb run validate`
  - TypeScript package invocation: `npx . validate`
  - Python package invocation: `uvx . validate`
- Keep canonical validation as `package validate`.
- Remove the `validate` script from `once/typescript/package.json`.
- Update docs and manual HTML to stop advertising bare validation shortcuts.

## Decisions from user

- Walter **should** gain `describe`.
- `lock`, `git-check`, `git-push`, and `unlock-any` are **not package-only**;
  they must also be documented/available for individual tool workflows.
- Remove **just** the bare `validate` shortcut. Do not broaden the task into
  removing every other existing compatibility shortcut.
- Do **not** add package-specific default lock metadata; only expose the verbs.
- Remove the `once/typescript` `validate` npm script.
- Update docs/manual HTML, not just CLI help/tests.

## Current findings

- The BigConfig SDK parser already recognizes the requested extra workflow verbs
  in all three SDKs:
  `lock`, `git-check`, `render`, `create`, `build`, `delete`, `validate`,
  `describe`, `exec`, `git-push`, `unlock-any`.
- Individual tool entry points already delegate their remaining args to SDK
  workflow parsing, so tool support for `lock`/`git-check`/`git-push`/`unlock-any`
  should mostly be documentation/help coverage plus tests/smoke checks.
- `once/clojure/src/clj/io/github/bigconfig_ai/once/cli.clj` is already mostly
  package-oriented, but still has a special bare `validate` branch and help text
  for it.
- `once/python/src/once/cli.py` and `once/typescript/src/cli.ts` expose bare
  `validate`; they also preserve some other bare package-step compatibility
  behavior. Per the decision above, remove bare `validate` only.
- `walter/{clojure,python,typescript}` currently rejects bare `validate` and has
  tests for that behavior, but it does not expose `describe` and does not list the
  new SDK verbs in help/package command sets.
- TypeScript `dist/` output is tracked in the TypeScript leaves, so source CLI
  changes need a build step that updates tracked `dist/src/cli.js`.

## Command model after implementation

### Once package steps

Supported under `package <step>...`:

- `validate` — pre-flight profile/tool/credential/image/SSH-agent checks.
- `describe` — post-provisioning report: providers, SSH reachability, deployed
  ONCE apps, and registry/update status.
- `build` — render all stages without applying/provisioning.
- `create` — provision and configure the full ONCE stack.
- `delete` — tear down the Tofu stages in reverse order.
- `lock` — acquire the SDK Git-tag lock.
- `git-check` — fail if the Git working directory/upstream check is not clean.
- `git-push` — run `git push` through the SDK workflow.
- `unlock-any` — force-release the computed SDK lock tag locally/remotely.

### Walter package steps

Supported under `package <step>...`:

- `validate` — pre-flight profile/tool/credential/Ansible-data checks.
- `describe` — report configured providers, compute SSH reachability, and the
  Walter workstation configuration summary.
- `build` — render Walter/Once stages without applying/provisioning.
- `create` — provision and configure the Walter workstation.
- `delete` — destroy the compute Tofu stage.
- `lock` — acquire the SDK Git-tag lock.
- `git-check` — fail if the Git working directory/upstream check is not clean.
- `git-push` — run `git push` through the SDK workflow.
- `unlock-any` — force-release the computed SDK lock tag locally/remotely.

### Tool workflows

All individual tools should continue accepting SDK workflow/exec sequences, and
help should explicitly say so:

- Once tools: `tofu`, `tofu-smtp`, `tofu-dns`, `tofu-smtp-post`, `ansible`,
  `ansible-local`.
- Walter tools: `tofu`, `ansible`, `ansible-local`.

Examples to show in help/docs:

- `tofu render tofu:init tofu:plan`
- `tofu git-check lock render tofu:init tofu:plan unlock-any`
- `ansible render -- ansible-playbook main.yml`

Do not add package-specific default `big-config.lock/lock-keys` or
`big-config.lock/owner`; callers can provide lock metadata in opts/env wrappers if
needed.

### Bare commands

- `help`, `--help`, `-h`, and no args print complete help.
- `package help`, `package --help`, and `package -h` should print complete help.
- Bare `validate` must fail with guidance to use `package validate`.
- Do not intentionally remove other existing compatibility shortcuts in this
  plan. If a leaf already rejects bare `build/create/delete/describe`, it may keep
  rejecting them. If a leaf already accepts some bare non-validate package verbs
  for compatibility, do not change that behavior unless required by tests.

## Walter `describe` design

Add a small Walter-specific `describe` namespace/module in all three languages,
mirroring the Once separation between pure report builder and workflow step.

Report builder shape:

- Clojure: `describe-report` and `describe` in
  `io.github.bigconfig-ai.walter.describe`.
- Python: `describe_report` / `describeReport` and `describe` in
  `walter.describe`.
- TypeScript: `describeReport` and `describe` in `src/walter/describe.ts`.

Inputs and seams:

- Accept `opts` and resolve params through `params/walter-opts` (or injected
  equivalent in tests) so `BC_PAR_*` overrides and Tofu output IP extraction are
  honored.
- Accept an injected command runner for SSH probes in tests.
- Reuse/copy the minimal Once describe helpers needed for:
  - provider summary;
  - compute target selection;
  - SSH reachability probe;
  - stable human-readable failure details.

Report contents:

- `profile`.
- `providers` with at least `compute` and `backend`.
- `compute` with `ip`, `user`, `running?`/`running`, and detail.
- `workstation` summary from `walter.ansible/data-fn` / `data_fn` / `dataFn`:
  - hosts;
  - sudoer;
  - users;
  - repo count and package count;
  - optional concise repo/package names if useful in printed output.
- `fatal-error?` / `fatalError` should normally be false. Unreachable SSH is a
  report detail, not a failing workflow step.

Printed report sections:

1. `Profile`
2. `Providers`
3. `Compute`
4. `Workstation`

Wire `describe` into Walter package workflows via the SDK describe hook:

- Clojure: add `::workflow/describe-fn describe/describe` in `walter/package.clj`.
- Python: add `bc_workflow.DESCRIBE_FN: describe` in `walter/package.py`.
- TypeScript: add `[WF_DESCRIBE_FN]: describe` in `walter/package.ts`.

Tests should use injected runners and params resolvers; do not require live SSH.
