## Goal

Create a developer-facing **BigConfig Package Spec**: a single standalone HTML page, in the same visual/manual style as `once-bigconfig/main/src/pages/manual.html`, for authors who want to implement a package that works with the `bc-pkg` launcher.

The spec should answer two questions:

1. **Launcher compatibility:** what must exist at the package repo root so `uvx bc-pkg ...` and `npx bc-pkg ...` can detect, install, initialize, re-enter, and run it.
2. **BigConfig package conventions:** how a package should expose lifecycle commands, profiles/params, templates/resources, and safe user-editable defaults, using `once` as the concrete reference package.

## Deliverables

- A single HTML page, modeled after `once-bigconfig/main/src/pages/manual.html`.

Key facts to capture:

- The launcher accepts either `<owner/repo@ref>` or a local path.
- `ref` is resolved to a 40-character commit SHA for GitHub targets.
- `GITHUB_TOKEN` is used when set.
- Target language is inferred from **exactly one** root manifest:
  - `deps.edn` â†’ Clojure
  - `package.json` â†’ TypeScript
  - `pyproject.toml` â†’ Python
- Root `run` is required.
- GitHub targets copy `run`; local targets symlink `run` with copy fallback.
- Re-entry reads BigConfig metadata from the generated manifest and refuses implicit switches/upgrades.
- Runtime commands are:
  - Clojure: `bb run ...`
  - TypeScript: `node run ...`
  - Python: `uv run python run ...`
- Clojure targets get generated `deps.edn` metadata plus `bb.edn` runtime deps.
- TypeScript targets get generated `package.json` dependency + `bigconfig` block.
- Python targets get generated `pyproject.toml` dependency + `[tool.bigconfig]` block.
- Local dev uses `:local/root`, `file:`, or editable `[tool.uv.sources]`.

Key facts to capture:

- A package can expose lifecycle verbs such as `validate`, `build`, `create`, `describe`, and `delete`.
- `run` should be safe to copy into a user's project and should carry editable default config/placeholders, not secrets.
- `run` should forward argv and pass a profile/options map into the package CLI/workflow.
- Profile params use kebab-case names matching template variables.
- Runtime overrides use `BC_PAR_*` env vars.
- Generated artifacts belong in `.dist/`; do not edit them as source.
- Templates live under packaged resources and must be included in the installable artifact.
- Template content commonly uses `<{ var }>` to avoid colliding with downstream `{{ ... }}` syntax.

## Proposed page outline

1. **Title / promise**
   - â€śBigConfig Package Specâ€ť
   - A package is a GitHub or local repo that `bc-pkg` can pin, initialize, and run through a root `run` file.

2. **Quick conformance checklist**
   - Exactly one root manifest.
   - Root `run` file exists.
   - Package name is declared for TS/Python.
   - Installable from GitHub SHA and local path.
   - Resources/templates are included.
   - `run` forwards args and contains safe defaults only.
   - Lifecycle commands are documented and tested.

3. **Launcher lifecycle**
   - First run with `<owner/repo@ref>`.
   - SHA pinning.
   - Manifest generation.
   - Dependency install.
   - Subsequent re-entry.
   - Local path mode.

4. **Repository root contract**
   - Required files by language.
   - Why multiple manifests are ambiguous.
   - What `bc-pkg` reads vs what it writes.

5. **Language-specific manifest contract**
   - Clojure: `deps.edn`, GitHub coord synthesized as `io.github.<owner>/<repo>`, generated `bb.edn`, Babashka compatibility.
   - TypeScript: root `package.json` with `name`, installable built JS or prepare/build strategy, package dependency by GitHub SHA/file path.
   - Python: root `pyproject.toml` with `[project].name`, installable wheel/sdist, `uv` compatibility, resources included.

6. **Root `run` file contract**
   - It is the user-facing copied/symlinked entrypoint.
   - It must be safe, editable, and free of credentials.
   - It should establish a default profile/options map and call the package CLI/workflow with forwarded argv.
   - Include three minimal examples based on `once`:
     - `#!/usr/bin/env bb` Clojure script.
     - `#!/usr/bin/env python3` Python script.
     - `#!/usr/bin/env node` TypeScript/ESM script.

7. **Command surface contract**
   - Recommended package verbs: `validate`, `build`, `create`, `describe`, `delete`.
   - Chaining behavior (`validate create`, `delete create`) when implemented through BigConfig workflows.
   - Tool-specific commands are package-defined, not launcher-defined.
   - The launcher forwards args; it does not understand package semantics.

8. **Profiles, params, and environment overrides**
   - Kebab-case params.
   - `BC_PAR_*` override shape.
   - Secrets stay outside source.
   - Destructive actions should require explicit opt-in safeguards.

9. **Templates and resources**
   - `src/resources/...` convention from `once`.
   - Include resources in package artifacts:
     - Clojure `:paths` includes `src/resources`.
     - TypeScript `files` includes `src/resources` and built `dist`.
     - Python hatch `force-include` or equivalent includes resources.
   - Directory/provider switching and content delimiters.

10. **Local development mode**
    - `bc-pkg ../package/path ...`
    - Refuses current directory.
    - Symlinked `run`.
    - Generated local deps per language.
    - TS local package needs built JS; Python is editable; Clojure uses `:local/root`.

11. **Reproducibility and updates**
    - GitHub ref resolves once to SHA.
    - No implicit upgrade.
    - Switching local/GitHub or repo/ref/path is a hard error.
    - How to intentionally re-initialize (new directory or remove generated manifest with care).

12. **Testing / acceptance matrix for package authors**
    - Test via both launchers where practical:
      - `uvx bc-pkg owner/repo@ref package validate`
      - `npx bc-pkg owner/repo@ref package validate`
      - local path equivalents.
    - Test first-run and re-entry.
    - Test `run` restoration by deleting generated `run` and re-running.
    - Test no credentials in copied files.
    - Test generated build artifacts are deterministic if the package claims parity.
