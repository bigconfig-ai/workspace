## Proposed behavior

### High-level behavior

1. `bb run git setup` walks the project table above in a stable order.
2. For each project group:
   - ensure the primary clone exists at the configured primary path;
   - create missing linked worktrees for the remaining branches;
   - validate existing primary/worktree paths;
   - fetch/pull all existing worktrees safely.
3. The command prints one row per primary/worktree plus a summary (`cloned`,
   `worktrees-created`, `updated`, `verified`, `dry-run`, `errors`).
4. Any clone/worktree/fetch/pull failure or invalid existing path exits non-zero.
5. `--root-dir DIR` applies to `setup` as it does to `report` / `commit`, so a
   user can bootstrap another workspace root with:

   ```sh
   bb run git setup --root-dir /path/to/bigconfig
   ```

6. `--dry-run` prints the clone/worktree/fetch/pull commands that would run and
   performs validation that can be done without mutation.

### Primary clone behavior

For each project primary:

- If the primary path does not exist:
  - create parent directories;
  - run `git clone --branch <primary-branch> <remote> <primary-path>`.
- If the primary path exists and is a git worktree:
  - validate `origin` matches the expected SSH remote;
  - validate the current branch equals the expected primary branch;
  - fail if validation does not pass.
- If the primary path exists but is not a git worktree, fail without overwriting
  anything.

### Linked worktree behavior

For each configured linked worktree:

- If the worktree path does not exist:
  - ensure the primary clone exists first;
  - fetch the target branch from origin;
  - run `git worktree add <worktree-path> <branch>` from the primary clone when the local branch already exists, or `git worktree add --track -b <branch> <worktree-path> origin/<branch>` when creating the local branch from the remote-tracking branch.
- If the worktree path exists and is a git worktree:
  - validate it belongs to the same expected remote;
  - validate its current branch equals the expected branch;
  - fail if validation does not pass.
- If the worktree path exists but is not a git worktree, fail without overwriting
  anything.

### Fetch/pull behavior for existing repos

For every validated existing primary/worktree, and for newly created
primary/worktrees after creation:

- run `git fetch --prune origin` from the primary clone for each project group;
- verify `git status --porcelain` is clean before pulling;
- run `git pull --ff-only origin <branch>` inside each primary/worktree.

Use `--ff-only` and an explicit `origin <branch>` so setup never creates merge commits or rewrites local history and does not require local branches to have upstream tracking configured.
If a worktree has dirty files or cannot fast-forward, `setup` exits non-zero after reporting the failing path.

## Remaining implementation notes

- `find -name .git` already discovers both `.git` directories and `.git` files;
  keep that behavior for `report` / `commit` so linked worktrees continue to be
  included.
- Prefer safe updates: no branch checkout, no merge commits, no reset, no clean,
  no deletion of unexpected paths.
- A project may need `git worktree prune` if a previous failed/manual setup left
  stale worktree metadata; the first implementation can fail with a clear Git
  error rather than pruning automatically.
