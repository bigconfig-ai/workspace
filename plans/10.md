# Goal

Get rid of `ONCE_PROFILE_JSON` and make the Python/TypeScript run-file shape
match the Clojure solution: the `run` file defines the default profile map/object
and invokes the CLI `main` with that profile directly.

# Desired shape

Clojure already has the desired pattern:

```clojure
(def default-profile {...})
(cli/main* *command-line-args* default-profile)
```

Python should become conceptually:

```python
DEFAULT_PROFILE = {...}
main(sys.argv[1:], DEFAULT_PROFILE)
```

TypeScript should become conceptually:

```ts
const DEFAULT_PROFILE = {...};
main(process.argv.slice(2), DEFAULT_PROFILE);
```

No profile should be passed through `ONCE_PROFILE_JSON`.

# Review notes

- The Clojure implementation already avoids an environment bridge because the
  `run` script requires the CLI namespace and calls `cli/main*` directly.
- The current Python and TypeScript implementations use `ONCE_PROFILE_JSON`
  because their `run` files spawn the packaged CLI in a child process. Removing
  the env var means the run files must stop using the packaged CLI as an opaque
  subprocess and instead call an importable CLI function.
- Python already has `main(argv=None, opts=None)` after the previous change, so
  it can be called directly once the package is importable in the interpreter
  running `run`.
- TypeScript needs `main` exported from `src/cli.ts` and the compiled
  `dist/src/cli.js`, plus a main-module guard so importing the CLI does not also
  execute it immediately.
- This touches the launcher contract for Python: `python run ...` with the
  system Python will not be able to import the package installed by `uv sync`
  unless either:
  - the launcher runs the script through uv, e.g. `uv run python run ...`; or
  - `run` re-execs itself through `uv run python run ...`; or
  - callers are expected to use the virtualenv Python directly.
- TypeScript can likely keep `node run ...` because Node resolves imports from
  the initialized directory's `node_modules` after `npm install`.

# Resolved decisions

1. Change the Python launcher/execution path so direct imports work.
2. Manual Python usage should also work through the direct-import path; update
   the run file / invocation contract accordingly instead of relying on
   `ONCE_PROFILE_JSON`.
3. For TypeScript, use JavaScript/direct imports if that makes the implementation
   simpler. It is acceptable for the run file to import the built JavaScript CLI
   entrypoint rather than preserving the old subprocess wrapper shape.
4. Remove `ONCE_PROFILE_JSON` support entirely.
5. Hardcoding the ONCE package import name (`once`) in these ONCE run files is
   acceptable.

# Proposed implementation plan

1. Python ONCE:
   - Remove `ONCE_PROFILE_JSON` handling from `once/python/src/once/cli.py`.
   - Keep/ensure `main(argv=None, opts=None)` supports an injected profile.
   - Change `once/python/run` to import `once.cli.main` and call it with
     `DEFAULT_PROFILE`.
   - Make the copied Python `run` file direct-import safe by ensuring it executes
     inside the uv environment when needed, or by changing the launcher to call
     it through uv.

2. TypeScript ONCE:
   - Export `main` from `once/typescript/src/cli.ts`.
   - Add a main-module guard so importing the CLI does not execute it.
   - Remove `ONCE_PROFILE_JSON` handling from the TypeScript CLI.
   - Change `once/typescript/run` to import the JavaScript CLI entrypoint and
     call `main(process.argv.slice(2), DEFAULT_PROFILE)` directly.
   - Rebuild `dist/` so installed package binaries and import paths are current.

3. Launchers:
   - Update both launcher implementations so Python targets are executed through
     the uv-managed environment, e.g. `uv run python run ...`, after `uv sync`.
   - Keep TypeScript targets as `node run ...` unless direct imports require a
     different command.

# Acceptance criteria

- `ONCE_PROFILE_JSON` is not used by any ONCE `run` file.
- Python and TypeScript `run` files define the default profile and directly call
  their CLI `main` function with it.
- The run-local profile is still used by package and individual-tool commands.
- Untouched templates still report `REPLACE_ME` validation errors instead of
  using packaged sample profiles.
- bc-pkg initialized directories still work for Clojure, Python, and TypeScript
  targets.
- No generated `.dist/` files are edited.
